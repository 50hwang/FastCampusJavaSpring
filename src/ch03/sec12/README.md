# ch03.sec12 인터페이스는 왜 쓰는가?

    연결점 : 사람과 TV 사이를 연결해주는 리모콘과 같은 역할

> 사실상 개발 실무에서 인터페이스 개념이 피부에 와닿는다.

## 인터페이스가 하는 일

- 서버 프로그램 측에서 클래스나 프로그램이 제공하는 기능을, 클라이언트 프로그램(3rd party)을 향해 명시적으로 선언
    - 일종의 클라이언트 코드와의 약속 : 클래스나 프로그램이 제공하는 명세(specification)
    - 클라이언트 프로그램은 인터페이스에 선언된 메소드 명세만 보고 이를 구현한 클래스를 사용할 수 있다.
      - 즉, 인터페이스로부터 구현된 세부 코드까지 일일이 보지 않아도 무방하도록 한다.
 
- 어떤 객체가 하나의 인터페이스 타입이라는 것은, 그 인터페이스가 제공하는 모든 메소드를 구현했다는 의미다.
    - 즉, 인터페이스가 제공하는 메소드들 중 단 하나라도 미구현된 메소드가 존재할 경우, 그 객체는 해당 인터페이스 타입이 아니다.
 
- 다형성 <- 인터페이스를 구현한 다양한 객체
    - 특정 인터페이스로부터 파생된 다양한 클래스들 중, 클라이언트 프로그램이 어떤 클래스를 사용하더라도, 동일한 `interface` 명세를 따르기 때문에, 사용 방법이 바뀌지 않는다.
    - ex) `JDBC` 인터페이스 : Java에서 `Connection`, `Statement` 등의 인터페이스를 사용하여 db를 연결하여 다룰 때, Oracle, MySQL, MS-SQL 등에서 저마다 다양한 `jar` 형태의 라이브러리를 사용할 수 있다.
      - 하지만 사용자 입장에서 클라이언트 프로그램을 구현할 때에는, `jar`를 일일이 뜯어보면서 다 알려고 할 필요도 없이, 인터페이스만 참고하여 사용하면 된다.
